# Настройка обновления Elo с Faceit (бесплатно, только Firebase + GitHub)

Как это работает: пользователь жмёт «Обновить Elo с Faceit» в своём профиле →
сайт ставит в его документе `users/{uid}` флаг `faceitUpdateRequested: true` →
раз в ~20 минут на GitHub (бесплатный GitHub Actions) запускается скрипт,
который находит всех с этим флагом, стучится в Faceit Data API и пишет
`faceitElo`/`faceitLevel`/`faceitUrl` обратно в Firestore. Ключ Faceit нигде,
кроме секретов GitHub-репозитория, не хранится — в код сайта он не попадает
и утечь через «просмотр кода страницы» не может.

Это **не мгновенно** (до ~20 минут ожидания), зато полностью бесплатно и не
требует привязки банковской карты (в отличие от Firebase Cloud Functions,
которым нужен платный план Blaze, пусть даже с бесплатной квотой).

Ничего из этого не нужно повторять при каждом деплое сайта — только один раз.

---

## Шаг 1. Получить ключ Faceit Data API

Он у вас уже есть (из кабинета разработчика Faceit, https://developers.faceit.com/).
Просто держите его под рукой для шага 3. Ключ выглядит как длинная строка
вида `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`.

## Шаг 2. Создать сервис-аккаунт Firebase

Это нужно, чтобы скрипт на GitHub мог писать в Firestore от имени доверенного
серверного процесса (в обход публичных правил безопасности сайта).

1. Откройте [Firebase Console](https://console.firebase.google.com/) → ваш проект.
2. ⚙️ (шестерёнка) рядом с «Project Overview» → **Project settings**.
3. Вкладка **Service accounts**.
4. Убедитесь, что выбран **Node.js**, нажмите **Generate new private key**.
5. Подтвердите — скачается файл `<project-id>-firebase-adminsdk-xxxxx.json`.
   **Никому не показывайте этот файл** — он даёт полный доступ к базе данных.

## Шаг 3. Добавить секреты в GitHub-репозиторий

1. Откройте репозиторий сайта на GitHub → **Settings** → в левом меню
   **Secrets and variables** → **Actions**.
2. Нажмите **New repository secret** и добавьте два секрета:

   | Name | Value |
   |---|---|
   | `FACEIT_API_KEY` | ваш ключ Faceit Data API из шага 1 (просто строка) |
   | `FIREBASE_SERVICE_ACCOUNT` | **всё содержимое** JSON-файла из шага 2, целиком, как есть (это валидный JSON-объект — просто откройте скачанный файл текстовым редактором, скопируйте всё и вставьте) |

3. Сохраните оба.

## Шаг 4. Проверить, что GitHub Actions включены

Обычно включены по умолчанию. Если нет:
**Settings → Actions → General → Actions permissions** → выбрать
«Allow all actions and reusable workflows» → Save.

## Шаг 5. Проверить работу вручную (не ждать 20 минут)

1. Вкладка **Actions** в репозитории → слева воркфлоу **Faceit Elo sync**.
2. Кнопка **Run workflow** (справа) → **Run workflow** ещё раз для подтверждения.
3. Через ~10–20 секунд обновите страницу — появится запись запуска.
   Кликните на неё → **sync** → там будет лог: сколько заявок найдено,
   для кого обновилось, для кого была ошибка (например, неверный ник).

Если лог зелёный (успех) — всё настроено, дальше воркфлоу будет сам
срабатывать каждые 20 минут и подхватывать новые заявки от пользователей.

---

## Что делать, если что-то не работает

- **`FACEIT_API_KEY не задан`** — секрет не сохранился или неправильное имя
  (должно быть ровно `FACEIT_API_KEY`, без пробелов).
- **`FIREBASE_SERVICE_ACCOUNT не задан`** — то же самое для второго секрета.
- **Ошибка при разборе JSON сервис-аккаунта** — при копировании файла в
  секрет что-то потерялось (например, вставилось не всё содержимое). Скачайте
  ключ заново на шаге 2 и вставьте файл целиком.
- **`Игрок с таким ником не найден на Faceit`** — пользователь ошибся в
  написании ника в профиле; это нормальная ситуация, скрипт просто помечает
  этого пользователя и идёт дальше, не ломая обновление остальных.
- **`превышен лимит запросов (429)`** — Faceit API временно ограничил
  частоту запросов; при следующем прогоне (через 20 минут) заявка будет
  обработана автоматически, флаг `faceitUpdateRequested` для неё сохранится.

## Про бесплатность

- GitHub Actions: 2000 бесплатных минут/месяц на приватном репозитории
  (для публичного — безлимитно). Один прогон занимает секунды-десятки
  секунд, 20-минутное расписание = ~2000+ прогонов в месяц с огромным
  запасом.
- Firebase: используется только Firestore (план Spark, бесплатный), сервис-
  аккаунт — это просто способ авторизации, отдельно не тарифицируется.
- Cloud Functions/Blaze не используются вообще.
